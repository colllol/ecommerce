# Complete RBAC (Role-Based Access Control) Web Application - Detailed Prompt

## Project Overview
Build a full-stack Role-Based Access Control (RBAC) web application that demonstrates enterprise-level authentication and authorization using a permission-based security model. The system allows administrators to define roles, assign permissions to roles, and assign roles to users, creating a flexible and scalable access control system.

---

## Technical Stack Requirements

### Backend:
- Node.js with Express.js framework
- MySQL database (relational data model)
- JWT (JSON Web Token) for stateless authentication
- bcrypt for password hashing (10 salt rounds)
- CORS enabled for cross-origin requests
- Environment variables via dotenv

### Frontend:
- React 19 with functional components and hooks
- Vite as build tool and dev server
- React Router DOM for client-side routing
- Bootstrap 5 for responsive UI design
- Axios for HTTP requests
- Context API for global state management

## Database Schema Design

Create a MySQL database with the following tables:

### 1. users table:
- `id` (INT, PRIMARY KEY, AUTO_INCREMENT)
- `username` (VARCHAR(50), UNIQUE, NOT NULL)
- `password` (VARCHAR(255), NOT NULL) - bcrypt hashed
- `email` (VARCHAR(100))
- `created_at`, `updated_at` (TIMESTAMP)

### 2. roles table:
- `id` (INT, PRIMARY KEY, AUTO_INCREMENT)
- `name` (VARCHAR(50), UNIQUE, NOT NULL)
- `description` (VARCHAR(255))
- `created_at` (TIMESTAMP)

### 3. permissions table:
- `id` (INT, PRIMARY KEY, AUTO_INCREMENT)
- `name` (VARCHAR(100), UNIQUE, NOT NULL)
- `description` (VARCHAR(255))
- `created_at` (TIMESTAMP)

### 4. user_roles table (many-to-many junction):
- `user_id` (INT, FOREIGN KEY в†’ users.id, ON DELETE CASCADE)
- `role_id` (INT, FOREIGN KEY в†’ roles.id, ON DELETE CASCADE)
- PRIMARY KEY (user_id, role_id)

### 5. role_permissions table (many-to-many junction):
- `role_id` (INT, FOREIGN KEY в†’ roles.id, ON DELETE CASCADE)
- `permission_id` (INT, FOREIGN KEY в†’ permissions.id, ON DELETE CASCADE)
- PRIMARY KEY (role_id, permission_id)

## Core Features Implementation

### 1. Authentication System:
- User registration with automatic "User" role assignment
- Login with JWT token generation (1-hour expiration)
- Password validation using bcrypt comparison
- Token-based session management stored in localStorage
- Profile endpoint to retrieve current user data with roles/permissions

### 2. Authorization Middleware:
- `verifyToken`: Validates JWT from Authorization header
- `checkRole(...roles)`: Grants access if user has ANY specified role
- `checkPermission(...permissions)`: Grants access if user has ANY specified permission
- Helper functions: `getUserPermissions(userId)`, `getUserRoles(userId)`

### 3. User Management (CRUD):
- GET all users with their assigned roles
- GET user by ID with roles and permissions
- POST create user (hashes password, assigns default User role)
- PUT update user (selective field updates, re-hash password if changed)
- DELETE user (cascades to user_roles)
- POST assign roles to user (replaces existing roles)

### 4. Role Management (CRUD):
- GET all roles with user count and permissions list
- GET role by ID with permissions and users list
- POST create role
- PUT update role
- DELETE role (cascades to role_permissions, user_roles)
- POST assign permissions to role (replaces existing permissions)

### 5. Permission Management (CRUD):
- GET all permissions with role count
- GET permission by ID with roles list
- POST create permission
- PUT update permission
- DELETE permission (cascades to role_permissions)

## API Endpoints Specification

### Authentication Routes (`/api/auth`):
- `POST /register` - Register new user
- `POST /login` - Login (returns token, user, roles[], permissions[])
- `GET /profile` - Get current profile (requires JWT)

### Users Routes (`/api/users`):
- `GET /` - Get all users (permission: VIEW_USERS)
- `GET /:id` - Get user by ID (permission: VIEW_USERS)
- `POST /` - Create user (permission: CREATE_USER)
- `PUT /:id` - Update user (permission: EDIT_USER)
- `DELETE /:id` - Delete user (permission: DELETE_USER)
- `POST /:id/roles` - Assign roles (permission: EDIT_USER)

### Roles Routes (`/api/roles`):
- `GET /` - Get all roles (permission: VIEW_ROLES)
- `GET /:id` - Get role by ID (permission: VIEW_ROLES)
- `POST /` - Create role (permission: CREATE_ROLE)
- `PUT /:id` - Update role (permission: EDIT_ROLE)
- `DELETE /:id` - Delete role (permission: DELETE_ROLE)
- `POST /:id/permissions` - Assign permissions (permission: EDIT_ROLE)

### Permissions Routes (`/api/permissions`):
- `GET /` - Get all permissions (permission: VIEW_PERMISSIONS)
- `GET /:id` - Get permission by ID (permission: VIEW_PERMISSIONS)
- `POST /` - Create permission (permission: CREATE_PERMISSION)
- `PUT /:id` - Update permission (permission: EDIT_PERMISSION)
- `DELETE /:id` - Delete permission (permission: DELETE_PERMISSION)

## Frontend Architecture

### Context API (AuthContext):
- Global state: `user`, `roles[]`, `permissions[]`, `isAuthenticated`
- Methods: `login()`, `logout()`, `hasPermission()`, `hasRole()`, `updateUser()`
- Persistent storage: token and user data in localStorage

### Protected Route Component:
- Wraps protected pages
- Checks for required permission via `hasPermission()` prop
- Redirects to login if not authenticated
- Shows "Access Denied" if missing permission

### Pages:
1. **Login** - Username/password form, demo accounts display, password visibility toggle
2. **Dashboard** - Sidebar navigation, stats cards, user info display
3. **Users** - User table, CRUD modals, role assignment dropdown
4. **Roles** - Role table, CRUD modals, permission assignment with multi-select
5. **Permissions** - Permission table, CRUD modals, color-coded badges by type

### Routing Structure:
```jsx
/login в†’ Login page
/dashboard в†’ Dashboard layout with nested routes
  / (index) в†’ DashboardHome (stats overview)
/users в†’ Users management (requires VIEW_USERS)
/roles в†’ Roles management (requires VIEW_ROLES)
/permissions в†’ Permissions management (requires VIEW_PERMISSIONS)
```

## Key Implementation Patterns

### 1. Permission Checking Flow:
```
User Login в†’ JWT Token в†’ Store in localStorage
         в†“
Request with Bearer Token в†’ verifyToken middleware
         в†“
checkPermission('VIEW_USERS') в†’ Query user_roles в†’ role_permissions
         в†“
Grant/Deny Access (403 Forbidden if denied)
```

### 2. Password Hash Generation:
```javascript
const saltRounds = 10;
const hashedPassword = await bcrypt.hash(plainPassword, saltRounds);
// Store hashedPassword in database
```

### 3. Role-Permission Query:
```sql
SELECT DISTINCT p.name FROM permissions p
INNER JOIN role_permissions rp ON p.id = rp.permission_id
INNER JOIN user_roles ur ON rp.role_id = ur.role_id
WHERE ur.user_id = ?
```

### 4. JWT Token Generation:
```javascript
const token = jwt.sign(
  { id: user.id, username: user.username },
  process.env.JWT_SECRET,
  { expiresIn: process.env.JWT_EXPIRE || '1h' }
);
```

### 5. Protected Route Pattern:
```jsx
<Route path="/users" element={
  <ProtectedRoute permission="VIEW_USERS">
    <Users />
  </ProtectedRoute>
} />
```

### 6. Middleware Protection Pattern:
```javascript
router.get('/', 
  verifyToken, 
  checkPermission('VIEW_USERS'), 
  userController.getAllUsers
);
```
